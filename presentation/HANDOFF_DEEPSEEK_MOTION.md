## Lead Review Verdict

The lead reviewer inspected the site with code review and Chrome screenshots for desktop, scrolled hero, route views, mobile, and `prefers-reduced-motion: reduce`.

Required fixes:

1. Reduced motion is currently incomplete.
   - In reduced-motion mode the page still enables Lenis smooth scrolling, GSAP scroll-scrub behavior, custom cursor, magnetic hover, parallax, and long reveal effects.
   - This is the highest-priority accessibility bug.

2. Hero scroll motion is too heavy.
   - The hero text is pushed away too far while scrolling.
   - Opacity drops too low, making the content feel like it is being thrown off the page.
   - Scroll scrub lag is too floaty.

3. Reveal, route, and hover motion is too slow or too strong.
   - Reveals use large vertical offsets and long durations.
   - Split text rotates too much and can stagger for too long.
   - Route transitions jump too far.
   - Magnetic hover and card hover move elements more than needed.

These are decisions from the lead reviewer. Do not re-litigate them.

## Patch 1: Reduced Motion Must Disable Motion Systems

Edit:

```text
presentation\site.js
presentation\src\css\08-anim.css
```

### JavaScript requirements

Add one shared reduced-motion flag near the top of `site.js`, after `const root=document.documentElement;`:

```js
const reduceMotion=matchMedia('(prefers-reduced-motion:reduce)').matches;
```

Use this flag consistently.

In the boot path, do not initialize Lenis or GSAP ScrollTrigger motion when `reduceMotion` is true.

Current behavior to change:

```js
await loadFirst(CDN.gsap);
await loadFirst(CDN.st);
await loadFirst(CDN.lenis);
gsap.registerPlugin(ScrollTrigger);
lenis=new Lenis({lerp:0.09,wheelMultiplier:1,smoothWheel:true});
```

Required behavior:

```js
if(reduceMotion){
  bindNativeScroll();
  gsapReady=false;
  return;
}
```

Then keep the existing GSAP/Lenis path for normal motion users.

Also ensure these systems are skipped when `reduceMotion` is true:

```text
Lenis smooth scrolling
GSAP ScrollTrigger hero timeline
GSAP parallax loop for [data-parallax]
custom cursor cursor-on class and cursor animation loop
magnetic hover binding
count-up animation, or make count-up complete immediately
smooth back-to-top scrolling duration
```

If a function already checks `matchMedia('(prefers-reduced-motion:reduce)')`, replace or align it with the shared `reduceMotion` constant.

Specific checks:

- `buildScroll(view)` must return immediately if `reduceMotion` is true.
- `autoEnhance(view)` may still add classes, but should not add meaningful parallax when `reduceMotion` is true.
- The custom cursor setup must not add `cursor-on` when `reduceMotion` is true.
- Magnetic hover handlers must not be bound when `reduceMotion` is true.
- If counters animate in JS, set final values immediately in reduced motion.

### CSS requirements

In `presentation\src\css\08-anim.css`, extend the reduced-motion block so all hidden reveal elements become visible and all major transform motion is removed:

```css
@media(prefers-reduced-motion:reduce){
  html{scroll-behavior:auto!important}
  .r,
  .reveal-lines .ln,
  .split-chars .ch,
  .mask-up{
    transition:none!important;
    transform:none!important;
    opacity:1!important;
    clip-path:none!important;
  }
  [data-parallax],
  .hero .label,
  .hero h1,
  .hero .sub,
  .hero .desc,
  .hero .row,
  .hero .stats,
  .pintrack,
  .pinpanel{
    transform:none!important;
  }
  #cursor,
  #cursor-ring{
    display:none!important;
  }
}
```

Do not leave any reveal element stuck at `opacity:0` under reduced motion.
